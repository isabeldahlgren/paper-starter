# proof-engineering — Implementation Plan

An autonomous research engine for pure mathematics. Input: papers from the user's research field. It executes `PROMPT.md` against each paper, iterates, and only interrupts the user when a result has independently passed **both** a taste gate and a correctness gate. Optimize for quality of output, not volume. A yield of one reviewable result per ~10 papers is success.

This document is the spec. Implement it exactly; where it is silent, choose the simplest option.

## Design principles

1. **Filesystem is the database.** No server, no queue system. Each paper gets a run directory; a `state.json` records the stage. The orchestrator is resumable and idempotent: re-running it continues unfinished runs.
2. **Prompts are files.** Every stage is a markdown prompt file in `prompts/`, executed via headless Claude Code (`claude -p "$(cat prompts/X.md)" ...`). Tuning the system means editing prompts, not code.
3. **Verifier independence.** Referees run in fresh contexts, never see the generator's reasoning or confidence — only the finished note. Adversarial framing: the referee's job is to *find the error*, not to confirm.
4. **Honest claim labels.** Every claim is tagged: `theorem-complete-proof` | `theorem-sketch` | `conjecture-with-evidence` | `computation` | `speculation`. Only `theorem-complete-proof` items enter the correctness gate as theorems; a note may still pass overall on the strength of a well-evidenced conjecture, but the labels must survive refereeing.
5. **The user's time is the scarcest resource.** False positives (bad results reaching review) are 10x worse than false negatives (good results archived). All thresholds err toward rejection. Every rejection is archived with reasons so the pipeline is auditable.

## Directory layout

```
proof-engineering/
  PROMPT.md              # the research brief (exists)
  PLAN.md                # this file
  orchestrate.py         # single orchestrator script
  config.toml            # models, budgets, thresholds, referee backend
  prompts/
    explore.md           # stage 2: wraps PROMPT.md with output-format contract + pre-commit literature check
    self_check.md        # stage 3: generator-side adversarial pass + counterexample search
    repair.md            # stage 3b: applies self_check's listed minor fixes (fixes.md) to the note
    novelty.md           # stage 4a: prior-art search (arXiv/web)
    taste_referee.md     # stage 4b: independent taste/relevance scoring
    correctness_referee.md  # stage 5a: independent line-by-line refereeing
                         # (stage 5b, the Lean track, is mechanical — aristotle CLI, no prompt file)
  taste/
    accepted/            # user-approved results (few-shot calibration for taste gate)
    rejected/            # user-rejected results, with her stated reason
  inbox/                 # user drops PDFs / arXiv IDs (one .txt with ID per line) here
  runs/<paper-slug>/
    paper/               # source (prefer arXiv LaTeX over PDF text extraction)
    state.json           # {stage, attempts, timestamps, verdicts}
    note.tex             # the research note (LaTeX, amsart class, abstract + upfront main result + novelty statement)
    result.json          # structured claims with labels, statements, proof status
    referee/             # taste + correctness reports, novelty search log
    lean/                # stage 5b: per-claim problem files + downloaded Aristotle Lean output
  review/<paper-slug>/   # symlink or copy of runs that passed both gates
  archive/<paper-slug>/  # everything else, with rejection.md stating why
```

## Pipeline stages

### 1. Intake
Watch `inbox/`. For each PDF or arXiv ID: create `runs/<slug>/`, fetch the paper (for arXiv IDs, download the LaTeX source — far better than PDF extraction; fall back to `pdftotext`), write `state.json` with `stage: explore`. Move the inbox item into the run.

### 2. Explore (generator)
Execute `prompts/explore.md`, which embeds `PROMPT.md` verbatim and adds the output contract: produce `note.tex` (LaTeX, `amsart` class — AMS-article style, one line per paragraph, brief abstract, main result(s) stated clearly up front, an explicit paragraph on what is novel relative to the source paper) and `result.json` listing every claim with its label, precise statement, and proof status. The generator has web access (`WebSearch`/`WebFetch`) and must run a short literature sanity check on its intended main result *before* writing the note (and again on any pivot): a rediscovered classical result should cost a few searches at the decision point, not the full explore + self-check spend before the novelty gate catches it. The generator may use Python/SymPy/Sage for computations and must include source code for any experiment in the run directory. Bounded: max 5 conjecture-repair iterations, then write up the strongest correct statement reached.

### 3. Self-check (cheap kill)
Fresh context, same or cheaper model. Two jobs: (a) adversarially re-derive each `theorem-complete-proof` claim and hunt for gaps; (b) **counterexample search** — for every universally quantified claim that admits finite/computable instances, write and run code testing small cases. A single counterexample kills the claim; the run loops back to stage 2 once (with the counterexample as feedback), then archives if it fails again. This is the highest-value check per dollar: numerics beat second opinions.

Minor defects below `gap-found` (prose contradicting the note's own table, a mislabeled script output) go into `self_check/fixes.md` as mechanical edit instructions. On a passing self-check, a single cheap **repair** invocation (`prompts/repair.md`) applies exactly those fixes before any referee sees the note — otherwise defects the self-check already diagnosed reach the correctness gate and trigger a full explore loop. Repair is best-effort and guarded: the orchestrator re-lints (`latexmk`, `result.json` schema, claim ids/labels/proof_status unchanged) and restores the pre-repair files on any failure.

### 4. Taste gate
- **4a Novelty:** search arXiv and the web for each surviving main claim. If **every** substantive claim is known, archive as `rejected: known` (log the references — still useful to the user as reading, but not review-worthy). If only some claims are known, the run has salvageable content: loop back to stage 2 with the known-vs-surviving breakdown as feedback (demote known results to cited background, rebuild the note around the surviving claims), archiving only when explore attempts are exhausted. The generator may read `referee/novelty_report.md` on that retry — independence protects referees from the generator, never the reverse.
- **4b Taste referee:** independent model, fresh context, sees only the paper + the note. Scores 1–5 on: relevance to the source paper, depth (does it need a real idea, or is it a one-line corollary of a standard theorem?), naturality of the question, and strength of the statement. Include the contents of `taste/accepted/` and `taste/rejected/` as few-shot calibration — **this is how the system learns the user's taste over time.** Pass requires ≥4 on depth and no score below 3. Explicit referee question: "Would a strong journal referee call this a routine exercise?"

### 5. Correctness gate (both tracks run; track A is mandatory)
- **5a LLM referee:** strongest available model (configurable; ideally a different provider than the generator — set in `config.toml`). Fresh context, sees only `note.tex` + `result.json`. Produces a line-by-line referee report; verdict per claim: `correct` / `fixable-gap` / `wrong` / `cannot-verify`. Require **two independent passes** (different seeds/contexts) both returning `correct` on every load-bearing claim. `fixable-gap` loops back to stage 2 once; anything else archives.
- **5b Lean track (bonus evidence, not a blocker):** Lean 4 formalization via the `aristotle` CLI (Harmonic's cloud autoformalization/proving service), not a self-hosted Mathlib build. Implemented as a `lean` stage between the correctness gate and review handoff, and strictly **non-gating**: the run has already passed both gates when it arrives, so no Aristotle outcome can archive it — the only exit is review. Because a proof can take hours, submission is **asynchronous**: one project per `theorem-complete-proof` claim via `aristotle formalize <file>` *without* `--wait` (the file is self-contained: instructions + the claim statement + the full note for definitions); the run then sits at the stage across orchestrator invocations, each tick polling with `aristotle tasks` (never `aristotle show`, which blocks while a task runs) and downloading finished projects. Outcome per claim lands in `result.json` as `formalization`: `proof-formalized` (Aristotle `COMPLETE` — machine-checked proof of the faithful statement), `statement-formalized` (`COMPLETE_WITH_ERRORS` — partial progress, never counted as a proof), or `not-formalizable` (failure / out-of-budget / `lean_timeout_minutes` exceeded, with a best-effort `aristotle cancel` to free the account's concurrent-task slot). A `proof-formalized` claim is marked ✓✓ in the review summary. Costs no Anthropic tokens and is exempt from `max_usd_per_paper` parking; requires `ARISTOTLE_API_KEY` + the CLI on PATH (`uv tool install aristotlelib`), and the stage is skipped (straight to review) without them or with `lean_enabled = false`.

### 6. Review handoff
Runs passing gates 4 and 5a land in `review/<slug>/` containing: `SUMMARY.md` (one page: the statement, why it's interesting in one paragraph, verification evidence table, links to referee reports and Lean files), `note.tex` compiled to PDF (`latexmk -pdf` must succeed — treat as a lint), and all supporting code. Notify the user (macOS `osascript` notification or a daily digest file; keep it simple). **After the user reviews:** she moves the result's summary into `taste/accepted/` or `taste/rejected/` with a one-line reason — closing the taste feedback loop.

## Orchestrator (`orchestrate.py`)

Single Python file, stdlib only. API keys (`ANTHROPIC_API_KEY`, `ARISTOTLE_API_KEY`) are read from a gitignored `.env` at repo root (template: `.env.example`; loaded at startup by a stdlib `load_dotenv()`, with already-set environment variables taking precedence) so cron invocations need no shell-profile setup. Responsibilities: scan `inbox/` and `runs/`, advance each run one stage per invocation via `subprocess` calls to `claude -p` (with `--permission-mode` and `--allowedTools` scoped per stage — referees get read-only + web; only explore/self-check/repair/lean stages get write/execute; explore additionally gets web for its literature check), enforce budgets, write `state.json` after every transition. Run it under `cron` or a `/loop`; it must be safe to invoke concurrently (lock file per run).

`config.toml` keys: `generator_model`, `self_check_model`, `taste_model`, `referee_model` (+ optional `referee_command` to plug in a non-Claude CLI for cross-provider refereeing), `max_runs_per_day`, `max_usd_per_run`, `lean_enabled`, `lean_timeout_minutes`, thresholds.

## Milestones (implement in order; each is shippable)

1. **M1 — Straight-through pipeline.** Intake → explore → self-check → archive/review, single paper, manual trigger, no gates beyond self-check. Verify the LaTeX note compiles and `result.json` validates against a schema.
2. **M2 — The gates.** Novelty search, taste referee with few-shot calibration hooks, double-pass correctness referee, rejection archiving with reasons. This is the heart of the system; spend the most care on the two referee prompts.
3. **M3 — Lean track.** Aristotle-backed asynchronous formalization stage after the correctness gate: per-claim submission, cross-invocation polling, per-claim timeout with cancel, outcomes recorded in `result.json` and the review summary. (Superseded the original plan of a pinned self-hosted Mathlib template with `lake build` as the check — Aristotle does the building and proving server-side, for free.)
4. **M4 — Autonomy.** Inbox watching, cron/loop scheduling, budget enforcement, notifications, and the `taste/` feedback loop.
5. **M5 — Self-improvement loop.** Each finished run should make later runs slightly better, with a human in the loop for anything that mutates the pipeline itself:
   - *Metrics first.* A `--stats` report computed mechanically from `review/` + `archive/` state histories defines "better": cost per reviewable result, per-stage pass/kill rates, spend lost to novelty-gate rediscoveries, explore attempts per run. No improvement claim without a number moving.
   - *Lesson distillation (`retrospect`).* After every finalize, one cheap fresh-context invocation reads the run's trajectory (state history, rejection reason, referee reports) and distills **at most one transferable, falsifiable lesson** into `lessons.md`, injected into future explore prompts. "On papers whose main theorem is an inequality, check the polynomial/function-field analogue in Lang before proving" qualifies; "be more careful about novelty" does not. The file is capped (fixed token budget) and every lesson cites the run it came from, so adding a lesson can mean merging or evicting one, and a bad lesson is auditable and deletable.
   - *Guarded prompt evolution.* When the same failure shape recurs (e.g. three archives at the same gate for the same kind of reason), retrospect may draft an edit to a `prompts/` file — written to `proposals/` as a diff with its evidence, applied only by the operator. Prompts are the system's genome; unsupervised self-modification risks optimizing the gates instead of the mathematics (the referee prompts especially must never be editable by anything the generator's output can influence).
   - The `taste/` corpus (M2/M4) is already the taste-gate half of this loop; M5 extends the same feedback pattern to the generator and to pipeline efficiency.

## Known weaknesses (do not paper over these)

- **LLM referees share failure modes with LLM generators.** Two models agreeing is much weaker evidence than one counterexample search or one Lean proof. That is why stage 3 runs code and stage 5b exists. Never present LLM-only verification as more than "survived independent refereeing".
- **Research-level formalization is usually out of reach.** Expect `not-formalizable` to be the common Lean outcome on genuinely interesting results; that is fine — the statement-level check still catches real errors.
- **Taste is the weakest gate on day one.** It becomes good only through the `taste/` corpus. Early on, expect the user to reject reviewable results; every rejection with a reason makes the gate stronger.

## Current status (updated 2026-07-08)

**M1 (straight-through pipeline) is built and validated on real content.** `arxiv-2505.11846` (a CNN-identifiability paper) went end-to-end through explore → self_check and produced a genuinely good result: a closed-form fiber-size formula that turns out to be Sperner's antichain bound, connecting the source paper's open "how singular" question to classical combinatorics. Self-check caught two real exposition gaps without treating them as errors and correctly left one claim `conjecture-with-evidence`/`unchecked` with a precise explanation. That run currently sits in `runs/arxiv-2505.11846/` reset to `stage: explore, attempts: 0` (paper source cached under `paper/`, no need to refetch) — its own note/result.json were cleared when the output format changed and it has not been re-run since.

**M2 (novelty + taste + correctness gates) is built but has never completed a real end-to-end run.** Pipeline order is now `explore → self_check (+repair) → novelty → taste → correctness (2 blind passes) → review/archive`. Referee isolation is real, not just prompted: `make_referee_dir()` copies only the allowed files (paper + note.tex + a stripped result.json with `proof_status`/`self_check_notes` removed) into a subdirectory and runs `claude -p` with that as `cwd` and no `--add-dir`, so referees cannot read self-check reports, prior referee passes, or each other — confirmed with a dummy-data smoke test (no LLM cost). `taste/accepted/` and `taste/rejected/` (the few-shot calibration corpus, see `build_taste_calibration()`) are still empty by the user's choice; the taste gate currently judges by the stated rubric alone.

**Lessons from the first real gated run (`arxiv-2408.04569`, archived 2026-07-08 at the novelty gate, $9.51).** The explore stage spent $5.20/53 turns proving a clean sharp bound that the novelty referee then correctly identified as the textbook "Fermat's Last Theorem for polynomials" corollary of Mason–Stothers (Lang, *Algebra*, 3rd ed., 1993, p. 195) — while three of the note's four claims (an explicit $k=4$ construction, a sharp-threshold conjecture with exact-arithmetic evidence, and a parity observation about the source paper's induction) survived the literature check and were discarded with the run. Separately, self-check had found three concrete minor exposition errors that nothing downstream could fix. Three changes landed in response, all validated offline (monkeypatched `run_claude`, scratch `ROOT`):
- **Explore now has web access and a mandatory pre-commit literature check** (`EXPLORE_TOOLS` adds `WebSearch WebFetch`; `prompts/explore.md` requires a few searches on the intended main result before the note is written, plus a sentence in `PROMPT.md` making novelty an explicit optimization target). Rationale: the $9.51 spent on a rediscovery would have been a few searches at minute ten.
- **A partially-known novelty verdict loops back to explore instead of archiving** (`do_novelty` distinguishes all-known → `reject` from some-surviving → `fail_to_explore_or_archive` with the known/surviving breakdown as feedback; `fail_to_explore_or_archive` gained a `stage` parameter). The generator may read `referee/novelty_report.md` on the retry — isolation protects referees from the generator, never the reverse.
- **Self-check emits `self_check/fixes.md` for minor defects and a passing check triggers one guarded repair invocation** (`apply_self_check_fixes` + `prompts/repair.md`, `REPAIR_TOOLS = Read Edit Bash`, `self_check_model`). Best-effort: the orchestrator re-lints and additionally verifies claim ids/labels/proof_status are untouched (`claim_skeleton`), restoring the pre-repair `note.tex`/`result.json` on any failure (`fixes.md` renamed to `fixes.applied.md`/`fixes.failed.md`). Without this, defects the self-check has already diagnosed to the line reach the correctness referee, whose `fixable-gap` verdict triggers a full ~$5 explore rerun.

**Output format: LaTeX + `amsart`** (switched back from Typst + `unequivocal-ams` per direct user request on 2026-07-08). History: LaTeX/amsart was the original format, reverted to Typst per an earlier user request, and has now been reverted back to LaTeX/amsart — this is the second flip, so treat the format as settled only until told otherwise. The motivation this time was standard mathematical-writing conventions (italics via `\emph{}` for terms at their point of definition, inline math for symbols in prose) that are native to `amsart`/`amsthm` rather than needing hand-rolled theorem environments, plus a title-length fix (titles must be a concise phrase, not a summary sentence) and setting the author field to "Proof Engineer". `orchestrate.py`'s `compile_latex()` runs `latexmk -pdf -interaction=nonstopmode -halt-on-error note.tex` as the lint (replacing `compile_typst()`), cleaning up `.aux`/`.log`/`.fls`/`.fdb_latexmk`/etc. afterward so only `note.tex`/`note.pdf` remain in the run directory. `prompts/explore.md` now gives the model the full `amsart` preamble (theorem/lemma/proposition/corollary/conjecture in `\theoremstyle{plain}`, definition/example in `\theoremstyle{definition}`, remark in `\theoremstyle{remark}`, all sharing one counter via `[section]`/`[theorem]`), matching standard AMS numbering directly rather than needing a custom package. Verified by hand-converting the archived `arxiv-2408.04569` note to `note.tex` and confirming a clean `latexmk` compile. The note must still include: brief abstract, main result(s) stated within the first half page, and an explicit "what this note adds vs. the source paper" paragraph — all per direct user feedback, not in the original spec below.

**Known-fixed bug: models were backgrounding long computations and ending their turn early**, e.g. "I'll wait for the background computation to finish rather than poll it." Since each stage is a single one-shot `claude -p` invocation with no future turn, this silently lost all output (no `note.tex`/`result.json` ever written). Fixed by adding explicit "never background — no `&`, `nohup`, and do not use your Bash tool's `run_in_background` option" instructions to `prompts/explore.md`, `prompts/self_check.md`, and `prompts/repair.md` (the only stages with Bash access). Not yet re-validated end-to-end after this fix.

**Token/session cost (2026-07-07: mitigated in code; billing choice still the operator's).** `claude -p` with no `ANTHROPIC_API_KEY` authenticates via the operator's Claude subscription (OAuth) and shares its session-based rate limit with interactive usage. Observed on a real run (`arxiv-2408.17221`): a single `explore` attempt cost $2.25, ran 18 turns/700s, read 1.6M cached tokens, then hit `api_error_status: 429` "session limit"; the instant retry burned the last attempt and the run archived without ever reaching `self_check`. What changed on 2026-07-07:
- **429/session-limit/overload responses are now treated as transient** (`is_transient_failure`/`defer_stage`): the run pauses at its current stage with no attempt consumed and retries on a later orchestrator invocation. The failure mode that killed `arxiv-2408.17221` (archiving a run because of a rate limit) is gone.
- **Referee-stage operational failures retry instead of archiving** (`referee_stage_error`, `max_referee_attempts`), and a `verdict.json` that omits a required claim or garbles a verdict value is a stage error, never a silent pass (previously an empty correctness verdict would have sailed a run into `review/` — a direct violation of design principle 5).
- **Intake prunes harder**: beyond figures/`.sty`/`.bib`, it now strips `.bbl`/aux/build residue and full-line TeX comments before the first LLM call, directly attacking the 1.6M-cached-token reads.
- **Pipeline calls are isolated from the operator's Claude Code config**: `claude_extra_args` defaults to `--strict-mcp-config` (personal MCP servers no longer load into — or leak into — every stage call); `--bare` is the documented full-isolation option (requires `ANTHROPIC_API_KEY`; OAuth is not read in bare mode).
- Per-stage estimated cost on `sonnet`: explore $2-6, self_check ~$2, novelty $1-3, taste $1-2, correctness×2 $2-4 → ~$8-17/paper. For unattended operation the README recommends API-key billing; subscription use still works but the pipeline and interactive usage starve each other.

**M4 (autonomy) is now mostly in place:** inbox watching with `max_runs_per_day` enforced at intake (surplus IDs stay queued in the inbox `.txt`), stale-lock recovery (a crashed orchestrator no longer blocks a run forever), a mechanical `SUMMARY.md` written on review handoff, and a macOS notification (`notify` in config). `finalize()` now *moves* runs out of `runs/` (in-flight only) instead of copying. Cron scheduling is the operator's (one `orchestrate.py` invocation per tick).

**M3 (Aristotle/Lean track) is built and validated offline (2026-07-09).** Pipeline order is now `explore → self_check (+repair) → novelty → taste → correctness → lean → review/archive`. The `lean` stage is exactly the non-gating asynchronous design in stage 5b above: `do_lean()` submits one Aristotle project per `theorem-complete-proof` claim (`aristotle formalize` without `--wait` — the CLI logs `Project created: <id>` to stderr and exits 0 even on errors, so the orchestrator parses output, never trusts return codes), tracks per-claim state under `lean` in `state.json` (`pending → submitted → proof-formalized | statement-formalized | not-formalizable`), polls with `aristotle tasks` (**not** `aristotle show`, which blocks while a task runs — verified in the aristotlelib source), downloads and unpacks finished projects into `lean/<claim>/`, cancels and marks `not-formalizable` on `lean_timeout_minutes`, and treats Aristotle's "too many requests" concurrent-task cap as stay-pending rather than a failure (3 hard submit failures = `not-formalizable`, never an archive). The stage is exempt from `max_usd_per_paper` parking (it costs no LLM dollars) and collapses to a straight correctness→review handoff when `lean_enabled = false` or `ARISTOTLE_API_KEY`/the CLI is missing. Validated offline with a stub `aristotle` CLI reproducing the real one's observable behavior (31 assertions across submit/poll/download, partial results, timeout+cancel, 429 back-off, hard failure, disabled, budget exemption — no LLM or network calls). **Not yet validated against the real Aristotle service**, and `codex`/`referee_command = "codex exec"` remains a documented-but-unused cross-provider refereeing hook. **Also still true: no run has completed the full gate sequence end-to-end** — the next real validation step is draining one paper through novelty → taste → correctness (→ lean) on API billing.

**M5 (self-improvement loop) is specced above (2026-07-09, at the user's request) but not started.** Build order within M5 should be metrics → retrospect/lessons → prompt proposals; the metrics report is a pure-stdlib afternoon and everything else is unmeasurable without it.

# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`paper-starter` is an autonomous research pipeline for pure mathematics. It reads a paper, executes `PROMPT.md` against it via headless Claude Code, and iterates until a result independently passes both a taste gate and a correctness gate. It optimizes for quality over volume — a yield of one reviewable result per ~10 papers is considered success. `PLAN.md` is the authoritative spec (design principles, directory layout, pipeline stages, milestones); read it before making structural changes, since this file only summarizes it.

**Doc layout** (three audiences — keep them in their lanes when editing): `README.md` is the newcomer quickstart (kept deliberately short — what it is, requirements, quickstart, "the three files you edit"); `DOCS.md` is the customization/reference guide (full config table, how to add gates, cost & auth, subscription-vs-API, the Lean track); `PLAN.md` is the design spec. Deep rationale belongs in DOCS.md or PLAN.md, not in README.md or as prose essays in `config.toml`.

## Commands

- Run one step of the pipeline (intake + advance every non-terminal run by one stage): `.venv/bin/python3 orchestrate.py`
- Drain every run to a terminal state (review/archive), looping stages until nothing progresses: `.venv/bin/python3 orchestrate.py --drain`
- Advance only one run, ignoring all others: `.venv/bin/python3 orchestrate.py --only <slug>` (e.g. `arxiv-2505.11846`)
- Print where every run stands (stage, cost, park/rejection reason) without advancing anything: `.venv/bin/python3 orchestrate.py --status`
- Compile a note by hand: `latexmk -pdf note.tex` (run inside the relevant `runs/<slug>/` directory — `orchestrate.py` also does this as a lint after the explore stage)
- Python deps live in `.venv` (stdlib `orchestrate.py` itself has none; `sympy`/`mpmath` are there for generator-stage experiments): `.venv/bin/python3 -m pip install <pkg>`

There is no linter or build step beyond the LaTeX compile check baked into the `explore` stage. Orchestrator logic changes should be exercised offline (monkeypatch `run_claude` and point `orchestrate.ROOT` at a scratch directory) — the failure-path behavior below is load-bearing and cheap to test without any LLM calls.

## Architecture

**Filesystem is the database.** There is no server or queue. Each paper gets `runs/<slug>/` with a `state.json` recording `{stage, attempts, feedback, history}`. `orchestrate.py` is a single stdlib-only script that is resumable and idempotent — re-invoking it just continues unfinished runs — and safe to call concurrently via a per-run `.lock` file (see `run_lock`).

**Pipeline stages**, driven by `advance()` in `orchestrate.py`, one stage per invocation: `explore → self_check (+repair) → novelty → <gates> → correctness → lean → review|archive`. The stage order is **data-driven** (`stage_sequence()`/`next_stage()`): a fixed head (`explore, self_check, novelty`) and tail (`correctness, lean`) with a user-configurable band of **pass/fail gates** spliced in between, one per `[[gates]]` entry in `config.toml` (ships with just `taste`). Each stage (except intake/lean/finalize) shells out to `claude -p` with a prompt built from a file in `prompts/` (`explore.md`, `self_check.md`, `repair.md`, `novelty.md`, `taste.md`, `correctness_referee.md`, plus one `<gate>.md` per configured gate) plus `PROMPT.md` where relevant. **Tuning the system means editing prompts and the gate list, not code.**

- `explore` (generator, `Read Write Edit Bash WebSearch WebFetch`): produces `note.tex` (LaTeX, `amsart` class) and `result.json` (claims with `label` ∈ `{theorem-complete-proof, theorem-sketch, conjecture-with-evidence, computation, speculation}` and `proof_status` ∈ `{unchecked, verified, gap-found, counterexample-found}`). `prompts/explore.md` mandates a **literature sanity check** (a few web searches) on the intended main result *before* the note is written, so a rediscovered classical result costs minutes, not the full explore+self_check spend before the novelty gate catches it. Gated by `compile_latex()` and `validate_result_json()` before advancing — and these gates, not the CLI exit code, are authoritative: an invocation that dies (timeout, dropped connection) after freshly writing both `note.tex` and `result.json` is **salvaged** (`explore_artifacts_fresh()` checks mtimes against the invocation start, so stale leftovers from a prior attempt never false-salvage) and judged by the gates like any other attempt. `prompts/explore.md` tells the generator to write both artifacts early and improve them in place, precisely so a cut-off session leaves something salvageable. Explore gets its own longer wall clock via `explore_timeout_seconds`.
- `self_check` (cheap kill, fresh context): adversarially re-derives `theorem-complete-proof` claims and runs counterexample search. Any claim left `gap-found`/`counterexample-found` sends the run back to `explore` (via `fail_to_explore_or_archive`), archiving once `max_explore_attempts` is exhausted. Minor defects below `gap-found` (prose contradicting a table, mislabeled script output) go into `self_check/fixes.md`; on a passing check, `apply_self_check_fixes()` runs one cheap **repair** invocation (`prompts/repair.md`, `Read Edit Bash`) to apply them before any referee sees the note. Repair is best-effort and guarded: if the repaired note fails to compile, `result.json` breaks, or claim ids/labels/proof_status change, the pre-repair files are restored and the run proceeds unrepaired.
- `novelty` / `<gates>` / `correctness`: **referee stages**. Verifier independence is enforced by the filesystem, not just prompting — `make_referee_dir()` copies only the allowed files (`paper/` + `note.tex` + a `result.json` stripped of `proof_status`/`self_check_notes` via `strip_result_for_referee()`) into a fresh subdirectory (`novelty_view/`, `<gate>_view/`, `correctness_view/`) and runs `claude -p` with that as `cwd` and no `--add-dir`, so referees cannot see prior reasoning, self-check notes, or each other's verdicts.
  - `novelty` distinguishes total from partial rediscovery: if **every** substantive claim is `known` the run archives, but if any claim survives the literature check the run loops back to `explore` with the known-vs-surviving breakdown as feedback (demote known results to cited background, rebuild the note around the surviving claims). The generator side may read `referee/novelty_report.md` on that retry — independence constraints protect referees from the generator, not the reverse.
  - `<gates>` (`do_gate()`) are the generic **pass/fail** referee stages — `taste` is the one that ships, but the shape is reusable and the list is user-extensible. Each runs one isolated `claude -p` over paper+note and must write `verdict.json` `{"verdict": "pass"|"fail", "notes": ..., "scores"?: {...}}`; **pass** advances to the next stage (`next_stage()`), **fail** archives via `reject()` — or, if the gate sets `on_fail = "explore"`, loops back to explore to be rewritten. Operational/transient failures (bad or missing verdict, 429s) are handled exactly like the built-in referee stages (`referee_stage_error()`/`defer_stage()`, `max_referee_attempts`). The `taste` gate sets `calibration = true`, which appends few-shot examples from `taste/accepted/` and `taste/rejected/` via `build_taste_calibration()` — how the system is meant to learn the user's taste over time (currently empty by choice; the gate judges on the stated rubric alone).
    - **Adding a custom gate**: (1) copy `prompts/gate_template.md` to `prompts/<name>.md` and write the criteria + `verdict.json` contract; (2) add a `[[gates]]` block to `config.toml` (`name` required; optional `prompt`, `on_fail` ∈ `{archive, explore}`, `include_paper`, `tools`, `model`, `calibration`). Gates run in listed order between novelty and correctness — put cheaper/earlier-killing gates first. `validate_gate_config()` fails fast at startup on a missing prompt file, a reserved/duplicate name, or a bad `on_fail`. No orchestrator code changes are needed — the sequence, dispatch (`advance()`'s gate loop), view-dir cleanup (`finalize()` globs `*_view`), and `SUMMARY.md` gate lines are all derived from the gate list.
  - `correctness` requires **two independent passes** to both return `correct` on every load-bearing (`theorem-complete-proof`) claim; `fixable-gap` on all bad claims loops back to `explore` once, anything else archives.
- `lean` (stage 5b, `do_lean()`): Aristotle (Harmonic's cloud Lean prover) formalization of each `theorem-complete-proof` claim — **bonus evidence, never a gate**: the run has already passed both gates, no outcome can archive it, and the only exit is review. No `claude -p` and no Anthropic tokens; the stage shells out to the `aristotle` CLI **asynchronously** (`formalize` without `--wait` — proofs can take hours) and the run sits at `lean` across orchestrator invocations, each tick submitting/polling/downloading what it can (poll with `aristotle tasks`; `aristotle show` blocks while a task runs; the CLI exits 0 even on logged errors, so output is parsed, never return codes). Per-claim state lives under `lean` in `state.json`; outcomes land in `result.json` as `formalization` ∈ `{proof-formalized, statement-formalized, not-formalizable}` (`COMPLETE` → proof ✓✓ in `SUMMARY.md`; `COMPLETE_WITH_ERRORS` → statement only, never counted as a proof; timeout after `lean_timeout_minutes` → best-effort `aristotle cancel` + not-formalizable; Aristotle's "too many requests" concurrent-task cap → stay pending, no failure counted). Skipped (straight to review) when `lean_enabled = false` or `ARISTOTLE_API_KEY`/the CLI is missing, and **exempt from budget parking** (costs no LLM dollars). Test offline against a stub `aristotle` on PATH, same pattern as `run_claude` monkeypatching.
- `finalize()`: **moves** the run into `review/<slug>/` (passed both gates — with a mechanically assembled `SUMMARY.md` and a macOS notification) or `archive/<slug>/` (with a `rejection.md` stating why); `runs/` holds in-flight work only. Referee view dirs and the `.lock` are stripped from the finalized copy.

**Failure semantics** (three distinct kinds — do not conflate them when editing):
- *Transient* (429/session-limit/overload/empty credit balance/connection closed mid-response, detected by `is_transient_failure()`): `defer_stage()` leaves the run at its current stage with **no attempt consumed**; a later invocation retries. Never archive for these. (In explore, the salvage check runs first — a "failed" invocation that already wrote fresh, gate-passing artifacts advances instead of deferring.)
- *Budget parking*: every stage's spend (parsed from `total_cost_usd` in the `claude -p` output) accumulates in `state.json`'s `cost_usd`; `advance()` parks a run **before** launching a stage that would exceed `max_usd_per_paper`. Parked ≠ archived — raising the cap resumes the run where it stopped.
- *Operational* (invocation crash, missing/malformed/incomplete `verdict.json`): retried in place — `fail_or_archive()` for explore/self_check (per-stage `max_*_attempts`), `referee_stage_error()` for referee stages (`max_referee_attempts`). A verdict that omits a required claim id counts as malformed, never as a pass.
- *Substantive* (gap/counterexample found, known result, taste fail, wrong/cannot-verify): loops back to explore where recoverable (`fail_to_explore_or_archive()` — self_check gaps, partially-known novelty verdicts, unanimous fixable-gap correctness verdicts), otherwise archives via `reject()` with the reason.

**Config** (`config.toml`): every key is **optional** — `load_config()` starts from the `CONFIG_DEFAULTS` dict at the top of `orchestrate.py` and overlays whatever `config.toml` sets, so a minimal (or absent) config still runs. When adding a config key, add its default to `CONFIG_DEFAULTS` and its one-line comment to `config.toml`, and document it in the DOCS.md config table — `config.toml` holds `key = value  # short`, not prose essays. Keys: `author` (the `\author{}` on every note), per-stage model choice (`generator_model`, `self_check_model`, `taste_model` — the novelty search referee — and `referee_model`, the default for `correctness` and every gate), the **`[[gates]]` array** (the user-extensible pass/fail gate band described above; keys `name`/`prompt`/`on_fail`/`include_paper`/`tools`/`model`/`calibration`), `max_runs_per_day` (enforced at intake; surplus inbox items stay queued), `max_usd_per_run` (a `--max-budget-usd` cap per `claude -p` call — does **not** protect against Claude Pro session-limit exhaustion, only per-dollar cost), `max_usd_per_paper` (cumulative per-run cap; parks, never archives), `max_referee_attempts`, `stage_timeout_seconds` (plus `explore_timeout_seconds`, a longer wall clock for the one stage that runs experiments and writes the note), `claude_extra_args` (extra flags for every `claude -p` call; defaults to `--strict-mcp-config` to keep the operator's MCP servers out of pipeline calls; `--bare` gives full config isolation but requires `ANTHROPIC_API_KEY`), `lean_enabled`/`lean_timeout_minutes` (the Aristotle stage: free, non-gating, needs `ARISTOTLE_API_KEY` + `uv tool install aristotlelib`), `notify`, retry caps, and `venv_python`. **API keys** (`ANTHROPIC_API_KEY`, `ARISTOTLE_API_KEY`) live in a gitignored `.env` at repo root (`.env.example` is the template), loaded by `load_dotenv()` at the top of `main()` — already-set environment variables always win over `.env`.

**Directories**: `inbox/` (drop PDFs or a `.txt` of arXiv IDs, one per line — watched by `intake()`), `runs/<slug>/` (in-flight), `review/<slug>/` (passed both gates), `archive/<slug>/` (rejected, with reason), `taste/{accepted,rejected}/` (calibration corpus the user curates by hand after reviewing).

## Output convention: LaTeX, not Typst

Notes are written in LaTeX using the `amsart` document class with `amsthm`-declared environments (`theorem`, `lemma`, `proposition`, `corollary`, `conjecture` in `\theoremstyle{plain}`; `definition`, `example` in `\theoremstyle{definition}`; `remark` in `\theoremstyle{remark}`; all sharing one counter per section, standard AMS numbering — `prompts/explore.md` gives the model the exact preamble). This is the second reversal of this decision: LaTeX/amsart was the original format, reverted to Typst (`unequivocal-ams` template) per user request, and has now been switched back to LaTeX/amsart per user request, this time explicitly for standard mathematical-writing conventions (`\emph{}` italics for terms at their point of definition, inline math for symbols in prose) and to fix overlong titles and the author field. Per the top-level style guide (`~/CLAUDE.md`), LaTeX source is one line per paragraph and the style imitates a top mathematical journal — the target is a note submittable as-is. `prompts/explore.md` spells out the house standards: main results stated as numbered theorems in the introduction, an explicit contributions paragraph, a "Relation to prior work" discussion in scholarly prose (never narrating the literature search — no tool names or search dates in the note), citation grammar (no "[1]'s theorem" — name the authors and attach `\cite`), display math for load-bearing formulas/matrices, computational certificates presented in the text with script pointers relegated to remarks, and complete `\bibitem` entries. `self_check` reports violations of these standards in `fixes.md` so the repair pass corrects them before any referee sees the note. The `\author{}` field comes from `author` in `config.toml` (substituted as `{{AUTHOR}}` into the explore prompt).

## Known operational hazards

- **Never let a `claude -p` invocation background a long computation.** Each pipeline stage is a single one-shot call with no future turn — if the model defers to a background job (`&`, `nohup`, the Bash tool's `run_in_background`), the output is silently lost (no `note.tex`/`result.json` ever written). `prompts/explore.md`, `prompts/self_check.md`, and `prompts/repair.md` (the only stages with Bash access) explicitly forbid this — and because prompting alone demonstrably failed (a real run died on "Background tasks still running after 600s"), `run_claude()` also hard-disables the feature via `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1` (which removes `run_in_background` from the Bash tool *and* stops the CLI auto-backgrounding slow foreground commands) and caps the shutdown wait via `CLAUDE_CODE_PRINT_BG_WAIT_CEILING_MS`. These env vars are undocumented CLI internals — re-verify them (`grep -a` the CLI bundle for `BACKGROUND`) if a CLI upgrade changes backgrounding behavior.
- **Claude Pro session limits, not just dollar cost, can stall the pipeline.** With no `ANTHROPIC_API_KEY` set, `claude -p` authenticates via OAuth and shares a session-based rate limit with the user's interactive usage; `max_usd_per_run` does not protect against this. The orchestrator now treats 429/session-limit/overload as transient (run pauses at its stage, no attempt consumed) rather than archiving, but on a subscription the pipeline and interactive usage still starve each other. Check whether `ANTHROPIC_API_KEY` is set (shell environment *or* `.env` — `load_dotenv()` loads the latter at startup) before assuming a stalled run is a bug; conversely, subscription mode requires the key absent from **both** (`env -u` alone can't unset what `.env` re-adds).
- **Token cost lives mostly in the paper source.** `intake` prunes non-content files (figures, `.sty`/`.bib`/`.bbl`, build residue) and strips full-line TeX comments before the first LLM call — extend `PAPER_SOURCE_STRIP_EXTS`/`strip_tex_comments()` rather than prompting models to ignore junk.

# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`paper-starter` is a **triage-and-verification** pipeline for pure mathematics. Generation happens *outside* it: the user runs a strong model (e.g. GPT via `GPT-PROMPT.md`) in a chat window to produce candidate research notes — one folder per project, each with `note.tex` + `result.json`. The user drops those folders in `inbox/`, and this tool, on the user's Claude subscription, does the three things a chat window can't: **cross-check** each note with a *different* model (re-derive the proofs, check the literature, judge the taste), **formalize** the proved claims in Lean via Aristotle, and **rank** the survivors into a `PRIORITY.md` reading list. It optimizes for the quality of what reaches the user's eyes — the user's reading time is the bottleneck. `PLAN.md` is the authoritative design spec; read it before structural changes.

The value proposition is **cross-model independence**: because GPT writes the note and Claude referees it, the crosscheck is genuine independent verification, not the same-model self-review a single chat window is stuck with. That single fact is why the pipeline is worth having and why generation was deliberately moved out.

**Doc layout** (three audiences — keep them in their lanes when editing): `README.md` is the newcomer quickstart (kept deliberately short — what it is, requirements, quickstart, "the files you edit"); `DOCS.md` is the customization/reference guide (config table, the input-folder contract, how crosscheck decides, cost & auth, the Lean track); `PLAN.md` is the design spec. Deep rationale belongs in DOCS.md or PLAN.md, not in README.md or as prose essays in `config.toml`.

## Commands

- Advance every non-terminal run by one stage (cron-friendly): `.venv/bin/python3 orchestrate.py`
- Drain every run to a terminal state, then regenerate the reading list: `.venv/bin/python3 orchestrate.py --drain`
- Advance only one run: `.venv/bin/python3 orchestrate.py --only <slug>`
- (Re)generate `PRIORITY.md` from `review/` without advancing anything: `.venv/bin/python3 orchestrate.py --rank`
- Print where every run stands: `.venv/bin/python3 orchestrate.py --status`
- Compile a note by hand: `latexmk -pdf note.tex` (inside `runs/<slug>/`; `orchestrate.py` also does this as a lint at the start of crosscheck)
- Python deps live in `.venv` (`orchestrate.py` itself is stdlib-only; `sympy`/`mpmath` are there for the crosscheck referee to re-run computations): `.venv/bin/python3 -m pip install <pkg>`

There is no linter or build step beyond the LaTeX compile check. Orchestrator logic changes should be exercised offline (monkeypatch `run_claude`/`run_aristotle` and point `orchestrate.ROOT` at a scratch directory) — the failure-path behavior below is load-bearing and cheap to test without any LLM or network calls. A worked offline harness pattern (intake → crosscheck pass/fail → lean → finalize → rank with stubs) is the fastest way to validate a change.

## Architecture

**Filesystem is the database.** No server, no queue. Each candidate note gets `runs/<slug>/` with a `state.json` recording `{stage, attempts, cost_usd, lean, history}`. `orchestrate.py` is a single stdlib-only script that is resumable and idempotent (re-invoking continues unfinished runs) and safe to call concurrently via a per-run `.lock` file (`run_lock`).

**Pipeline stages**, driven by `advance()`, one stage per invocation: `intake → crosscheck → lean → review|archive`, plus a global `rank`. The stage flow is linear and short by design — the multi-stage generation scaffold this repo used to have was removed, because a good one-shot generation prompt in a chat window makes it redundant. **Tuning the system means editing `prompts/crosscheck.md`, `prompts/rank.md`, and `GPT-PROMPT.md` — not code.**

- `intake()` (`intake_folder`): ingests each `inbox/` subdirectory that has `note.tex` + `result.json`. Validates `result.json` (`validate_result_json` — non-empty claims, each with id/label/statement; `proof_status` accepted-if-present but not required, since it's GPT's self-assessment which the referee never sees). Copies the folder into `runs/<slug>/`; reads the `paper` field and, if it's an arXiv id (`arxiv_id_from_paper_ref`), fetches the source into `paper/` (`fetch_paper_for` → `fetch_arxiv_source` + `prune_paper_dir` + `strip_tex_comments`) so the referee can judge novelty/motivation. Best-effort: a note whose paper can't be fetched is still refereed. Malformed folders are skipped and moved to `inbox/.processed/`. `max_runs_per_day` throttles intake.
- `do_crosscheck()`: the **one referee stage** and the heart of the system. Runs a single isolated `claude -p` (`prompts/crosscheck.md`, model `referee_model`, tools `Read Write Bash WebSearch WebFetch`) over an isolated view dir and writes `verdict.json` covering correctness (per-claim `correct`/`fixable-gap`/`wrong`/`cannot-verify`), novelty (`novel`/`known`/`uncertain`), and taste scores, plus an overall `verdict: pass|fail`. **Verifier independence is enforced by the filesystem, not just prompting**: `make_referee_dir()` copies only `paper/` + `note.tex` + a `result.json` stripped to id/label/statement (`strip_result_for_referee`) into `crosscheck_view/` and runs with that as `cwd` and no `--add-dir`, so GPT's `proof_status`, `verification_report.md`, and audit notes are never visible. **Pass → lean; fail → archive** (`reject`) with the reasons. On top of the referee's verdict, `do_crosscheck` enforces two hard rules in code regardless of the overall verdict — fail if any `theorem-complete-proof` claim is worse than `correct`, or if every substantive claim is `known` — because those are the false positives that most waste the user's time. A verdict missing a load-bearing claim, or malformed, is an operational error (`referee_stage_error`, `max_referee_attempts`), never a silent pass. A `fixable-gap` rejection records exactly what's missing, so the user can hand that back to GPT and re-drop.
- `do_lean()` (`lean` stage): Aristotle (Harmonic's cloud Lean prover) formalization of the note — **bonus evidence, never a gate**: the note has already passed crosscheck, no outcome can archive it, the only exit is review. **One project per note** (not per claim): `aristotle submit "<prompt>" --project-dir <dir with note.tex>`, where the prompt is the editable `prompts/lean.md` (`read_lean_prompt`, default `LEAN_DEFAULT_PROMPT` — "formalise all results; report real proof errors in ERROR.md"). No `claude -p` and no Anthropic tokens; shells out to the `aristotle` CLI **asynchronously** (`submit` without `--wait` — proofs can take hours), the run sitting at `lean` across invocations, each tick submitting → polling (`aristotle tasks`; `aristotle show` blocks while a task runs) → downloading (`aristotle download` yields a `.tar.gz`, unpacked into `lean/solution/`). The CLI exits 0 even on logged errors, so output is parsed, never return codes. Single-project state under `lean` in `state.json`; the **note-level** outcome lands in `result.json` under `formalization` ∈ `{proof-formalized, statement-formalized, not-formalizable}` — `proof-formalized` only when `COMPLETE` **and** `scan_lean_output()` finds no `sorry` and no `ERROR.md`; `COMPLETE_WITH_ERRORS`, a remaining `sorry`, or a reported error all downgrade to `statement-formalized`; timeout after `lean_timeout_minutes` → best-effort `aristotle cancel` + not-formalizable; "too many requests" cap → stay pending, no failure. An `ERROR.md` (Aristotle found a real proof error) is copied to `lean/ERROR.md` and surfaced **loudly** (⚠️ banner atop `SUMMARY.md`, ⚠️ flag pushed to the top of `PRIORITY.md`) but stays non-gating. Skipped straight to review when `lean_enabled = false`, no `theorem-complete-proof` claim exists, or `ARISTOTLE_API_KEY`/the CLI is missing; exempt from budget parking. Test offline against a stub `aristotle` on PATH.
- `finalize()`: **moves** the run into `review/<slug>/` (passed crosscheck — with a mechanical `SUMMARY.md` from `write_review_summary` and a macOS notification) or `archive/<slug>/` (with a `rejection.md` whose text is framed as feedback to hand back to GPT). `runs/` holds in-flight work only; the referee view dir and `.lock` are stripped from the finalized copy (`finalize()` globs `*_view`).
- `do_rank()`: after a `--drain` (or on `--rank`), one `claude -p` (`prompts/rank.md`) reads a factual digest (`collect_reviewed`) of every `review/` note — title, claims, per-claim novelty, Lean status, taste scores — and writes `PRIORITY.md`, a ranked reading list weighing depth/strength, verification strength (Lean-proved > referee-only), and novelty, with a short "not recommended" section from `collect_rejected`. `write_priority_fallback` produces a deterministic ordering if the call fails, so `PRIORITY.md` is always written.

**Failure semantics** (three distinct kinds — do not conflate them):
- *Transient* (429/session-limit/overload/empty-credit/connection-closed, via `is_transient_failure()`): `defer_stage()` leaves the run at its stage with **no attempt consumed**; a later invocation retries. Never archive for these.
- *Budget parking*: every stage's spend (from `total_cost_usd`) accumulates in `state.json`'s `cost_usd`; `advance()` parks a run **before** launching a stage that would exceed `max_usd_per_paper`. Parked ≠ archived — raising the cap resumes it. `lean` is exempt.
- *Operational* (crash, missing/malformed `verdict.json`): `referee_stage_error()` retries in place up to `max_referee_attempts`. A verdict omitting a required load-bearing claim counts as malformed, never a pass.
- *Substantive* (referee `fail`, a broken proof, or all-known): archives via `reject()` with the reason.

**Config** (`config.toml`): every key is **optional** — `load_config()` overlays `config.toml` on the `CONFIG_DEFAULTS` dict at the top of `orchestrate.py`, so a minimal or absent config still runs. When adding a key, add its default to `CONFIG_DEFAULTS`, its one-line `key = value  # short` comment to `config.toml`, and a row to the DOCS.md config table. Keys: `referee_model` (crosscheck + rank — pick a *different family* than the generator), `max_runs_per_day` (intake throttle), `max_usd_per_run` (a `--max-budget-usd` cap per call — does **not** protect against subscription session-limit exhaustion), `max_usd_per_paper` (cumulative cap; parks, never archives), `max_referee_attempts`, `stage_timeout_seconds`, `claude_extra_args` (defaults to `--strict-mcp-config`; `--bare` gives full config isolation but requires `ANTHROPIC_API_KEY`), `lean_enabled`/`lean_timeout_minutes`, `notify`, `venv_python`. **API keys** live in a gitignored `.env` at repo root (`.env.example` is the template), loaded by `load_dotenv()` at the top of `main()` — already-set environment variables win.

**Directories**: `inbox/` (drop GPT project folders — a dir with `note.tex` + `result.json`), `runs/<slug>/` (in-flight), `review/<slug>/` (passed crosscheck), `archive/<slug>/` (rejected, with reason), `PRIORITY.md` (the reading list at repo root).

## Output convention: LaTeX, not Typst

Notes are LaTeX using the `amsart` document class with `amsthm`-declared environments. paper-starter does **not** generate notes, so it does not enforce this — `GPT-PROMPT.md` does (it hands the generator the exact `amsart` preamble and house style: `\emph{}` for terms at their point of definition, inline math for symbols in prose, main results stated up front, an explicit contributions paragraph, scholarly "relation to prior work", complete `\bibitem`s). Per the top-level style guide (`~/CLAUDE.md`), LaTeX source is one line per paragraph and the style imitates a top mathematical journal. `orchestrate.py`'s only interaction with the format is `compile_latex()` (`latexmk -pdf`), run at the start of crosscheck as a lint and to produce `note.pdf`; a non-compiling note is flagged but still refereed from source.

## Known operational hazards

- **Never let a `claude -p` invocation background a long computation.** Each stage is a single one-shot call with no future turn — if the model defers to a background job (`&`, `nohup`, the Bash tool's `run_in_background`), the output is silently lost. `prompts/crosscheck.md` (the only stage with Bash access) forbids it, and because prompting alone demonstrably failed once, `run_claude()` also hard-disables the feature via `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1` and caps the shutdown wait via `CLAUDE_CODE_PRINT_BG_WAIT_CEILING_MS`. These are undocumented CLI internals — re-verify (`grep -a` the CLI bundle for `BACKGROUND`) if a CLI upgrade changes backgrounding behavior.
- **Subscription session limits, not just dollar cost, can stall the pipeline.** With no `ANTHROPIC_API_KEY`, `claude -p` authenticates via OAuth and shares a session rate limit with interactive usage; `max_usd_per_run` does not protect against this. 429/session-limit/overload are treated as transient (pause, no attempt consumed). Subscription mode requires the key absent from **both** the shell environment and `.env` (`env -u` alone can't unset what `.env` re-adds). Because only refereeing runs here (not a full generation pipeline), the session pressure is far lighter than the old design imposed.
- **Token cost lives mostly in the source paper.** `fetch_paper_for` prunes non-content files (figures, `.sty`/`.bib`/`.bbl`, build residue) and strips full-line TeX comments before the referee call — extend `PAPER_SOURCE_STRIP_EXTS`/`strip_tex_comments()` rather than prompting the model to ignore junk.
